# Critical Review: 2026-05-21 Sprint

**Reviewer hat:** independent methodologist auditing the day's work.
**Standing question:** what got built that the project can rely on, vs. what got built fast and may need to be revised or rolled back?

This document is meant for the project maintainers to consult before publishing v0.4.1 / v0.5 announcements. It is intentionally critical. The day's work was productive; that's not the question. The question is whether productivity outran rigor.

**Solid (keep as-is):**

- LEADERBOARD Executive Summary uses declarative language ("headline findings") for what is genuinely a single-tool single-day pilot. Tone overclaims.
- fmd_report.py `partial_complete` override (timeout-with-files) deviates from RFC 0004 §3.1 strict precedence on operational grounds — unprincipled, should either change the RFC or remove the override.
- Task 05 specified hastily (terse-only, no verbose/casual). T01 had three variants; T05 has one. Inconsistent.

**Oversold (recommend tone-down on next commit):**
- "First organic timeout observation" — was actually the smoke harness's adapter-level 3600s timeout firing, not a tool-internal hang/timeout. Mis-labels harness-induced cap as tool-induced failure.
- "All-or-nothing pattern" — claim made with N=2-3 per condition, then partly retracted when T01 terse +NI run 3 produced PRS 218. Original claim was too strong.
- "PRS test-retest converges within 2 points" — true for N=2 on T04 only. With N=3+ the spread is 14-30 points. We've been pulling the optimistic number forward.
- "Phase transition discovery" framing in commit `733b433`. Stochastic variance from a small sample is not a phase transition.

**Active risks:**
1. Cost overrun: ~30 `claude -p` runs collected this session, each using subscription quota. Aggregate cost is non-trivial and was not budgeted.
2. Methodology drift: non-interactive suffix is being treated as "the fix" but it changes what is being measured (claude-code's behavior under a forced-execution prefix ≠ claude-code's behavior as users actually use it). Defensible only if explicitly disclosed in every report.
3. Tight coupling: 22 commits each touch multiple documents (README + LEADERBOARD + RFCs + tests). Hard to revert any single decision. Methodology and reporting documents drifting together rather than independently.

## Detailed review by artifact

### 1. RFC 0001 (Maintainability/Quality, Draft)

**Status:** Strongest piece of the day's work.
**Empirical case:** T03 wrong_artifact 1 file = Quality 0, real codebase 35 files = Quality 62. This is exactly the kind of evidence an RFC needs.
**Recommendation:** Keep. Open formal comment period. Cite the T03 case in the announcement. Don't bury it in §2.4.

### 2. RFC 0004 (Failure Mode Distribution, Draft)

**Status:** Conceptually right, but rushed to implementation.
**Concerns:**
- The taxonomy was published, then validated against 23 historical runs, then a reference implementation was shipped — all in the same day. Comment period hasn't happened.
- The `timeout` mode label is being applied to runs that were *killed by the smoke harness's 3600s wait*, not runs where the tool itself hung. This is a category mistake. Real `timeout` means the tool wouldn't have completed even given more time; harness-cap kills are different. The taxonomy needs a `harness_truncated` mode (or similar), and the existing `timeout` mode needs a stricter definition.
- `partial_complete` override in `fmd_report.py` deviates from the RFC's strict precedence. If the override is correct, the RFC needs revision; if the RFC is correct, the override should be removed. The current state — RFC says one thing, reporting tool does another — is the worst of both worlds.
- Wrong-artifact detection uses an extension-ratio heuristic (`>0.5 doc files`) rather than the LLM-as-judge specified in §3.3. Defer-to-future is fine *if* documented as a known gap in the implementation; right now it's just inconsistent.

**Recommendation:** Pause shipping the reference implementation as "production." Treat `harness/scoring/failure_mode.py` as a *draft* alongside the RFC. Add a `STATUS: pre-comment-period draft` comment to the top of the file. Revise the RFC to address the harness_truncated category. Schedule a real comment period (2 weeks per `rfcs/README.md`) before merging into v0.5.

### 3. Non-Interactive Suffix (v0.4.1 patch)

**Status:** Real discovery, but the framing needs work.
**What is real:**
- claude-code in `-p` mode does ask for clarification on many prompts (this is reproducible via `scripts/diag_claude_silent_decline.py`)
- An execution-cue suffix raises completion rates substantially
- The completion rate without the suffix was *not* 100% as N=1 runs implied

**What is overclaimed:**
- The suffix is presented as "the fix." It is *one* answer to a methodology choice that has multiple defensible positions:
  - **Pro-suffix:** measure the tool's *capability* not its *interaction politeness*
  - **Anti-suffix:** the agent's clarification-asking IS the user-facing behavior — measuring something else is measuring something else
  - **Compromise:** report both conditions side-by-side, never collapse to one
- The current LEADERBOARD already shows both no-NI and +NI rows, which is good. But the Executive Summary frames "+NI is the right way to run the benchmark" implicitly by foregrounding the +NI scores.

**Recommendation:** Either (a) treat NI vs no-NI as a permanent reported axis (right answer in my view), with both reported in every cycle; or (b) write an RFC that specifies which mode is the "official" benchmark and why. Don't leave it as "opt-in by default" — that's ambiguous.

### 4. fmd_report.py auto-discovery (operational tool)

**Status:** Useful, but with two methodological cheats:
- The "composite_prs is non-null + files ≥ 5 → complete" override bypasses the classifier entirely. Pragmatic, but it means the classifier is not actually the source of truth. The classifier's behavior diverges from the report's behavior.
- The "timeout + files ≥ 5 → partial_complete" override violates RFC 0004 §3.1 strict precedence.

**Recommendation:** Remove both overrides from `fmd_report.py`. The classifier should be the source of truth. If we want different behavior, fix the classifier (and the RFC). Document the divergence as a known issue requiring an RFC patch.

### 5. T05 spec (just added)

**Status:** Right idea, wrong execution.
**Concern:** T01 had verbose + casual + terse variants. T02-T04 have only terse. T05 has only terse. Per the prompt-variant-sensitivity finding documented in LEADERBOARD, this is exactly the lopsidedness the methodology says is problematic.

**Recommendation:** Don't ship T05 to a future cycle with only one variant. Either backfill verbose+casual for T02-T05 before any official cycle, OR commit to terse-only across all 10 tasks and document the prompt-variant test as a separate validation cycle.

### 6. Executive Summary on LEADERBOARD

**Status:** Useful for readability but tone overclaims.
**Concerns:**
- "Four headline findings" — but this is a single-tool single-day pilot. Headlines should come from cycles.
- "1.7× discrimination improvement" is real but specific to one case (T03). Generalizing it requires more cases.
- "All-or-nothing pattern" was qualified after T01 terse +NI run 3 came in, but the summary's confidence isn't fully dampened.

**Recommendation:** Re-frame as "Preliminary observations from the v0.4.1 pilot" rather than "Executive Summary." Soften "headline" → "candidate" or "preliminary." Add an explicit "what we'd need to confirm this" column to each finding.

### 7. 22 commits in one day, tight coupling

**Status:** Velocity outran direction.
**Concerns:**
- Each commit touches multiple documents (README + LEADERBOARD + RFC + sometimes tests/source).
- Reverting any single methodology decision now requires touching N files.
- Many commits land "for completeness" rather than "because a decision was made."

**Recommendation:** Going forward, separate (a) methodology-change commits from (b) data-collection commits from (c) reporting-update commits. Keep each commit narrow. Don't update the leaderboard *and* an RFC *and* the README in the same commit unless they truly form one atomic unit.

### 8. Data N is small

**Status:** Statistically thin.
- Per-condition N ranges 1-5 with median ~3.
- We've drawn conclusions (test-retest stability, all-or-nothing pattern, phase transitions) that genuinely require N=10+ per condition.

**Recommendation:** Restrict published findings to claims that hold at N=2-3 (i.e., "we observed X at least once"). Pull the more confident claims ("converges", "tends to", "pattern is") back until N=10+ exists.

## Specific concrete fixes recommended

These are small enough to do in the next session:

1. **Remove the two overrides from `fmd_report.py`.** Let the classifier be the source of truth. Document the divergence elsewhere.
2. **Add a `harness_truncated` mode to RFC 0004 §3.1** OR redefine `timeout` to exclude the case where the *harness* killed the tool (current behavior is the latter, mislabeled).
3. **Re-tone the Executive Summary.** Replace "headline findings" with "preliminary observations." Add a "confidence" column or footnote.
4. **Backfill verbose+casual variants for T02-T05** OR mark T01's verbose/casual as deferred to future cycles.
5. **Add a `STATUS: pre-comment-period draft` banner to `harness/scoring/failure_mode.py`** until RFC 0004 closes its comment period.
6. **Budget audit.** Total claude -p run cost this session ≈ $50-75. Should be acknowledged in any cycle reporting.

## What NOT to do

Things that would feel right but would be wrong:

1. **Don't roll back yesterday's work.** Methodology evolution is fine; the issue is tone and rigor, not direction.
2. **Don't add more dimensions/RFCs in the next session.** RFC 0001 (Quality) and RFC 0004 (FMD) are enough for v0.5. Adding a third would dilute focus.
3. **Don't launch more parallel batches** until the analytical disciplines (CIs, multiple-comparison correction, etc.) are wired into the reporting tools. More data is not the problem; more *interpreted* data is the problem.
4. **Don't publish a v0.4.1 announcement yet.** Wait until the above six concrete fixes are addressed.

## Strategic note

The day's productivity was real. The work is overwhelmingly load-bearing rather than overbuilt — Quality dim, FMD taxonomy, non-interactive suffix, diagnostic tooling, OSF template, failure_record.json forensic trail are all genuinely useful artifacts.

What outran rigor was the *tone* and *velocity of reporting*, not the underlying work. The fix is restraint in how the work is framed, not abandonment of the work itself.

A useful mental check before any future commit during a productive sprint: *"would I be comfortable defending this commit's claims under cross-examination at a methodology audit?"* For most of today's commits the answer is yes. For the ones flagged above, the answer is "with caveats" — and those caveats should be in the document, not in my head.
